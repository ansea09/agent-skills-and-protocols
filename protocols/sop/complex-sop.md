# Complex SOP

This SOP explains the complex checklist bundle.

## Purpose

Use the complex protocol when risk, ambiguity, evidence load, external action, multiple viewpoints, code blast radius, or user impact makes a short checklist insufficient.

## 10 Intake And Scope

Parse the user message into requests, questions, tasks, constraints, and artifacts. Bound the task before acting. State assumptions when proceeding without clarification. Ask a question only when a reasonable assumption would be risky.

## 20 Actors, Roles, Work

Identify who or what performs work. Codex, scripts, tools, agents, repositories, APIs, and the user may all act in different roles. Keep role, method, method description, plan, actual work, and output separate.

## 30 Sources And Evidence

Use source evidence only when it is appropriate for the claim. Prefer primary and expert sources. Reject sources that look AI-generated, SEO-driven, unowned, or detached from evidence. Current claims require freshness checks.

## 40 Reasoning And Answer

Write the main answer in the user's domain language. Keep viewpoints separate until reconciliation. Repair overloaded terms and vague relations. Do not invent facts. Label hypotheses. Keep code or file edits idiomatic to the repository.

## 50 Final Audit

Check consistency, temporal adequacy, source adequacy, possible alternative conclusions, instruction hierarchy, and answer quality. Include FPF and protocol commits, cache warnings, source choices, and residual risks.

## Output Shape

For complex tasks, summarize the executed phases in the engineering basis. Print the whole checklist only when the user asks for audit detail.
